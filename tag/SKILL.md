# Tag — raw 파일에 주제 태그 붙이기

raw/에 쌓인 파일들이 각각 무엇에 관한 것인지 **frontmatter `tags`** 로 분류하는 가벼운 트리아지 작업.
이 볼트의 `CLAUDE.md` §4(태그 규약)·§6(Tag 워크플로우)를 실행 절차로 구체화한 것. 충돌 시 `CLAUDE.md`를 따른다.

> [!important] 절대 원칙 (wiki 승격 아님)
> - raw 파일을 **그 자리에 그대로** 둔다. **이동·삭제·파일명 변경·본문 수정 전부 금지.**
> - 오직 frontmatter의 `tags` 필드만 수정/추가한다. 다른 메타데이터(`ingested`, `source`, `created` 등)는 건드리지 않는다.
> - 이 스킬은 wiki/ 페이지를 만들지 않는다. 그건 `ingest`의 일. tag와 ingest는 **독립**(태그만 달고 ingest 안 해도 되고, 그 반대도 됨).

핵심 기조: **묻지 말고 보여주기.** 막히지 않는 한 먼저 태그를 달아 보여주고, 교정은 사용자가 Obsidian에서 한다.

> [!tip] 권장 순서: tag 먼저 → 나중에 ingest
> raw에 파일이 수십 개 쌓이면, 먼저 `tag`로 전체를 빠르게 분류해 무엇이 있는지 조망한 뒤 가치 있는 것만 골라 `ingest`로 깊이 정리한다. ingest는 raw에 이미 달린 주제 태그를 소스 페이지 태그의 출발점으로 재사용할 수 있다.

## 대상 선정
- 사용자가 특정 파일을 지목하면 그것을 처리한다. **이미 주제 태그가 있는 파일이라도 지목하면 재태깅 대상**으로 삼는다(절차 4의 재태깅 규칙 적용).
- 지목이 없으면 `raw/`(및 `raw/articles/`)에서 **아직 주제 태그가 없는 파일**을 대상으로 삼는다.
  - 판단: `tags`가 없거나, `clippings` 같은 형식/자동 태그만 있고 **주제(topical) 태그가 없는** 파일 = 미태깅.
  - `ingested` 여부와 무관하게 태깅 가능(태그는 검색·분류용이라 ingest된 파일도 주제 태그가 없으면 대상).
- 여러 개면 목록을 먼저 보여주고 한 건씩 처리한다. 수십 개면 진행상황을 중간중간 보고.
- `README.md`, `_`로 시작하는 파일 등 소스가 아닌 파일은 제외.

## 절차

### 1. 관리 태그 레지스트리 로드 — `_tags.md`
- `_tags.md`를 읽어 현재 캐논 태그 어휘를 파악한다. 태그는 **이 목록에서 우선 선택**.

### 2. 파일 읽기 → 주제 파악
- 각 파일 본문을 읽고 "무엇에 관한 것인지" 파악. (이미지 위주면 핵심만 view)
- 형식(documentation/tutorial/article/paper/snippet)과 주제(도메인+구체 토픽)를 함께 판단.

### 3. 태그 선정
- **기존 태그 우선 재사용**. 동의어 신설 금지(`ros2` 있으면 `ros-2` 만들지 말 것).
- 영어 kebab-case 평면. 보통 **도메인 1개 + 구체 주제 2~5개** (총 3~6개).
- 필요한데 레지스트리에 없는 태그는:
  1. `_tags.md`의 알맞은 그룹에 `- \`태그\` — 한 줄 뜻` 으로 추가하고 `updated`를 오늘로,
  2. 그 태그를 파일에도 적용한다.
- 추측 남발 금지 — 파일을 실제로 설명하는 태그만.

### 4. frontmatter 수정 — raw 파일
- 기존 `tags`에 **병합**(append). 기존 태그(`clippings` 등)는 **지우지 않는다**. 이미 같은 태그가 있으면 **중복 추가하지 않는다(dedupe)**.
- 파일이 쓰던 YAML 스타일을 유지: 리스트형(`- "x"`)이면 리스트로 추가, 인라인형(`[a, b]`)이면 인라인으로 추가.
- frontmatter가 아예 없는 파일이면 맨 위에 `---` 블록을 새로 만들어 `tags`를 넣는다. (본문은 그대로 아래에 둔다)
- **frontmatter의 `tags`만 손댄다.** 다른 줄은 그대로.
- **재태깅(개별 문서)**: 사용자가 이미 태깅된 파일을 지목해 다시 요청하면 재태깅한다.
  - 기본은 여전히 **append**(누락된 주제 태그 보강) + dedupe.
  - 단, 사용자가 교정을 명시하거나("고쳐줘 / 정리해줘 / 이 태그 틀렸어 / 다시 달아줘") 명백히 부정확·중복인 태그가 있으면 **그 태그를 제거·교체**할 수 있다. 그 외 기존 태그는 보존.
  - 즉 자동(대량) 태깅은 append-only지만, **명시적 재태깅은 교정(remove/replace)까지 허용**한다. 무엇을 바꿨는지 보고에 적는다.

### 5. 로그 — `log.md`
- 맨 위에 엔트리 append:
  ```
  ## [YYYY-MM-DD] tag | <파일 N건>
  - <파일1>: [태그...]   <파일2>: [태그...]
  - 레지스트리 신규 태그: `x`, `y` (없으면 생략)
  ```

## 완료 보고 형식
- 🏷️ 태깅한 파일과 부여 태그 (파일별). 재태깅이면 제거/교체한 태그도 표기.
- 🆕 `_tags.md`에 새로 추가한 태그 (있으면)
- ⏭️ 건너뛴 파일 (이미 태깅됨 / 소스 아님)
- 🚀 **다음 단계 (항상 안내)** — 아래 참고.

### 🚀 다음 단계 안내 (보고 끝에 항상)
태깅을 마치면 ingest로 자연스럽게 이어지도록 다음 단계를 **명시적으로** 안내한다:
- 내용상 위키 편입 가치가 있는 파일을 골라 `[[파일]]`로 나열하고, `/ingest` 로 편입하라고 안내한다. (권장 순서 tag → ingest)
- 단순 메모·중복 등 ingest가 불필요한 파일은 태깅만으로 충분하다고 명시한다.
- 예시:
  > 🚀 다음 단계 — ingest 권장: `[[파일A]]`, `[[파일B]]`. `/ingest`로 위키에 편입하세요. (나머지는 태깅만으로 충분)

사용자는 Obsidian의 Tags 패널·Properties에서 결과를 보며 교정한다.
